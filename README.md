## 📋 نظرة عامة

نظام **ERP** متكامل يعمل بالكامل داخل المتصفح دون الحاجة لأي خادم خلفي (backend). يستخدم `localStorage` كقاعدة بيانات مدمجة ويوفر واجهة عربية كاملة من اليمين لليسار (RTL)، مع دعم الوضع الليلي والنهاري، ولوحات تحكم تفاعلية بالرسوم البيانية.

النظام مصمم خصيصاً لإدارة الدورة الإنتاجية الكاملة في شركات الرخام والجرانيت: من **استلام الكتل الخام** ← **القطع والتصنيع** ← **إدارة الجودة** ← **المخازن والألواح** ← **المبيعات والتصدير** ← **التحصيل والمدفوعات** ← **المحاسبة والتقارير المالية**.

🌐 **الموقع المباشر:** [https://yousefeid1.github.io](https://yousefeid1.github.io)

## 🗺️ تدفق العمليات الكاملة وربط الأقسام

يعمل النظام كدورة متكاملة تبدأ من المورد وتنتهي عند العميل وتعكس نفسها في المحاسبة. فيما يلي رسم تخطيطي مبسّط لتدفق البيانات بين الأقسام:

```
المشتريات (فاتورة مورد)
        ↓
المخازن — كتل خام (Blocks)
        ↓
التصنيع — القطع (Cutting) → الألواح (Slabs)
        ↓
مراحل التصنيع (20 مرحلة) + تكاليف المناولة
        ↓
إدارة الجودة (تصنيف A/B/C/D) + رفع صور
        ↓
كتالوج المنتجات النهائية
        ↓
المبيعات (فواتير + عروض أسعار + CRM)  OR  التصدير (أوامر تصدير)
        ↓
اللوجستيك — الشحنات والمستودعات
        ↓
المدفوعات والمتحصلات
        ↓
المحاسبة (قيود يومية + ميزان مراجعة + تقارير مالية)
```

**ربط الأقسام بالتفصيل:**

| من | إلى | طريقة الربط |
|----|-----|-------------|
| فاتورة شراء (مشتريات) | كتلة خام (مخازن) | رقم الكتلة `block_code` مدوّن في بند الفاتورة |
| كتلة خام → عملية قطع | دفعة قطع `cutting` | `block_id` / `block_code` مرتبط بكل دفعة |
| دفعة قطع | ألواح (slabs) | كل لوح يحمل `cutting_id` و `block_id` |
| ألواح + مراحل تصنيع | تكلفة الإنتاج | مراحل `manufacturing_stages` مرتبطة بـ `blockId` |
| الألواح | إدارة الجودة | درجة الجودة (A/B/C/D) تُضبط على `slab.grade` |
| منتج نهائي | فاتورة مبيعات | `product_id` في بنود الفاتورة، يُحدَّث `stock_qty` تلقائياً |
| عرض أسعار | فاتورة مبيعات | زر "تحويل لفاتورة" ينسخ البنود وينشئ فاتورة جديدة |
| فاتورة مبيعات | شحنة | `invoice_id` / `invoice_number` مدوّن في الشحنة |
| أمر تصدير | شحنة / مستودع | `export_order_id` مرتبط بالشحنة |
| مدفوعات/متحصلات | رصيد العميل/المورد | `party_id` + `party_type` → يُحدَّث `balance` |
| أي عملية مالية | قيود يومية | يُنشأ قيد محاسبي تلقائياً أو يدوياً بالقيد المزدوج |
| قيود يومية | ميزان المراجعة | مجموع مدين/دائن لكل حساب في `accounts` |
| ميزان المراجعة | P&L + ميزانية عمومية | تقارير مالية تُجمع الأرصدة تلقائياً |
| مراكز التكلفة | تكلفة المتر المصنع | توزيع التكاليف غير المباشرة على مراحل الإنتاج |
| سجل النشاط | كل العمليات | كل إضافة/تعديل/حذف يُسجَّل تلقائياً بالمستخدم والوقت |

## ✨ تفاصيل الوحدات والأقسام

### 1 · 📊 لوحة التحكم (`dashboard.js`)

لوحة مركزية تعرض ملخصاً فورياً لحالة الشركة من جميع الأقسام.

**المؤشرات الرئيسية (KPIs):**
- إجمالي المبيعات (محسوب من جداول `sales`)
- إجمالي المشتريات (من جدول `purchases`)
- إجمالي المتحصلات (من جدول `payments` نوع `receipt`)
- صافي الأرباح (إيرادات − تكاليف − مصروفات)
- نسبة النمو الشهري ▲/▼ لكل مؤشر

**الرسوم البيانية (Chart.js):**
- مبيعات شهرية — مخطط خطي
- توزيع المصروفات — مخطط دائري (Pie)

**الجداول الديناميكية:**
- أكثر المنتجات مبيعاً مع شريط تقدم بالنسبة المئوية
- تنبيهات المخزون المنخفض (المنتجات التي `stock_qty < min_stock`)
- آخر 5 فواتير مبيعات مع حالتها

**ربط لوحة التحكم بالأقسام:** تقرأ البيانات مباشرة من `DB.getAll('sales')`, `DB.getAll('purchases')`, `DB.getAll('payments')`, `DB.getAll('products')`, `DB.getAll('notifications')`.

### 2 · 🛒 المشتريات والموردين (`purchases.js`)

**إدارة الموردين:**
- بيانات المورد: الاسم، رقم التليفون، البريد، العنوان
- رصيد الحساب المتراكم (مجموع الفواتير − المدفوعات)
- تاريخ بدء التعامل

**فواتير المشتريات:**
- رقم الفاتورة (PUR-YYYY-NNN تلقائياً)
- تاريخ الفاتورة وتاريخ الاستحقاق
- بنود الفاتورة (وصف البند، الكمية، السعر، الإجمالي)
- حالة الفاتورة: `مسودة → مرسلة → مدفوعة جزئياً → مدفوعة بالكامل`
- دعم العملات: EGP, USD, EUR, GBP, SAR, AED
- طباعة الفاتورة بصيغة PDF

**الربط مع أقسام أخرى:**
- عند تسجيل فاتورة شراء بلوك، يُذكر رقم البلوك في وصف البند مما يربط المشتريات بالمخازن
- عند تسجيل دفعة للمورد في قسم المدفوعات، يُحدَّث رصيد المورد تلقائياً
- يمكن إنشاء قيد محاسبي يدوياً (مخزون مدين / البنك دائن) لكل فاتورة شراء

### 3 · 🪨 المخازن والكتل الخام (`cutting.js`)

#### أ) الكتل الخام (Blocks)
كل كتلة واردة تُسجَّل بالتفاصيل التالية:

| الحقل | الوصف |
|-------|-------|
| رقم الكتلة | كود تلقائي `BLK-YYYY-NNN` |
| نوع الحجر | رخام أبيض كراراني / جرانيت أسود مطلق / إلخ |
| مصدر الكتلة | إيطاليا / الهند / البرازيل / تونس / الصين / إلخ |
| الوزن (طن) | بالطن |
| الأبعاد (سم) | عرض × ارتفاع × طول |
| التكلفة | التكلفة الإجمالية بالجنيه |
| حالة الكتلة | `في المخزن ← تحت القطع ← تمت معالجته` |
| تاريخ الاستلام | |

**حالات الكتلة وتدفقها:**
```
in_stock → in_cutting → processed
```

#### ب) عمليات القطع (Cutting Batches)
كل دفعة قطع تُسجَّل بـ:

| الحقل | الوصف |
|-------|-------|
| رقم الدفعة | `CUT-YYYY-NNN` |
| الكتلة المقطوعة | `block_id` مرتبط بكتلة محددة |
| تاريخ القطع | |
| المشغّل | اسم العامل المسؤول |
| عدد الألواح الناتجة | إجمالي الألواح |
| الألواح درجة A | |
| الألواح درجة B | |
| الهالك (م²) | الكميات التالفة |
| نسبة الهالك (%) | محسوبة تلقائياً |
| حالة الدفعة | `جارٍ العمل ← مكتملة` |

**الربط:** عند اكتمال دفعة القطع، تُنشأ ألواح تلقائياً في جدول `slabs` مرتبطة بنفس الكتلة والدفعة.

#### ج) الألواح (Slabs)

| الحقل | الوصف |
|-------|-------|
| رقم اللوح | `SLB-NNN` |
| مرتبط بكتلة | `block_id` / `block_code` |
| مرتبط بدفعة قطع | `cutting_id` |
| نوع الحجر | موروث من الكتلة |
| درجة الجودة | A / B / C / D (تُعيَّن من إدارة الجودة) |
| الأبعاد (سم) | عرض × ارتفاع |
| السماكة (سم) | |
| المساحة (م²) | محسوبة تلقائياً |
| الحالة | `في المخزن ← مباع ← تحت التصنيع` |
| الصور | حتى 5 صور مضغوطة (JPEG 70%) |

### 4 · ⚙️ مراحل التصنيع (`manufacturing.js`)

يتتبع النظام **20 مرحلة إنتاجية** مدعومة:

| # | المرحلة | # | المرحلة |
|---|---------|---|---------|
| 1 | نشر الرخام | 11 | شطف الحواف |
| 2 | نشر الجرانيت | 12 | طمبل (Tumbling) |
| 3 | تلميع | 13 | سنفرة |
| 4 | معالجة بالريزن | 14 | حرق |
| 5 | معالجة بالإيبوكسي | 15 | تصنيع حوض / مغسلة |
| 6 | تشطيب وتر جيت (Water Jet) | 16 | تصنيع درج |
| 7 | كالبريشون (Calibration) | 17 | تصنيع وتر جيت مخصص |
| 8 | فلقة (Flaming) | 18 | تغليف |
| 9 | هوند (Honing) | 19 | تحميل وشحن |
| 10 | براشد (Bush Hammering) | 20 | مرحلة مخصصة |

**بيانات كل مرحلة:**

| الحقل | الوصف |
|-------|-------|
| الكتلة | `blockId` مرتبط بكتلة خام |
| نوع المرحلة | من القائمة أعلاه |
| التاريخ | تاريخ تنفيذ المرحلة |
| الكمية المعالجة | مع وحدة القياس (م² / م³ / طن / لوح / قطعة) |
| التكلفة المباشرة | تكلفة الماكينة والطاقة |
| تكلفة العمالة | عدد العمال × ساعات العمل × 8 ج.م/ساعة |
| تكلفة المواد | مواد مستهلكة (أقراص قطع، مواد لصق، إلخ) |
| تكلفة النقل | نقل داخلي |
| درجة الجودة الناتجة | درجة أولى / ثانية / ثالثة / مرفوض |
| ملاحظات | |

**الربط مع مراكز التكلفة:** يمكن تخصيص التكاليف غير المباشرة (كهرباء، إيجار، إلخ) على مراحل الإنتاج بأسس التوزيع: المساحة / ساعات التشغيل / التكلفة المباشرة.

### 5 · ✦ إدارة الجودة (`quality.js`)

**درجات الجودة ومعاملات التسعير:**

| الدرجة | التعريف | معامل السعر الافتراضي |
|--------|---------|----------------------|
| A (درجة أولى) | جودة ممتازة | 100% |
| B (درجة ثانية) | جودة جيدة | 75% |
| C (درجة ثالثة) | جودة مقبولة | 50% |
| D (مرفوض) | غير صالح للبيع | 0% |

- معاملات التسعير قابلة للتعديل من واجهة المستخدم وتُحفظ في `grade_multipliers`
- السعر النهائي للوح = السعر الأساسي × معامل درجة الجودة

**رفع الصور:**
- حتى **5 صور** لكل لوح
- الصور أكبر من 1 MB تُضغَط تلقائياً إلى JPEG بجودة 70% وعرض أقصى 800px
- تُخزَّن الصور كـ Base64 في localStorage

### 6 · 💰 المبيعات (`sales.js`)

#### أ) إدارة العملاء
- بيانات العميل: الاسم، التليفون، البريد، العنوان، نوع العميل، المنطقة
- رصيد الحساب (إجمالي الفواتير − المتحصلات)
- تاريخ أول تعامل

#### ب) فواتير المبيعات
**دورة حياة الفاتورة:**
```
مسودة → مرسلة → مدفوعة جزئياً → مدفوعة بالكامل → ملغية
```

**تفاصيل الفاتورة:**
- رقم الفاتورة: `INV-YYYY-NNN`
- العميل (مرتبط بـ `customer_id`)
- بنود الفاتورة: المنتج (`product_id`) × الكمية × السعر × الإجمالي
- حساب الضريبة: نسبة الضريبة من الإعدادات (افتراضي 14%)
- دعم العملات: EGP, USD, EUR, GBP, SAR, AED مع سعر الصرف
- مندوب المبيعات: `salesperson_id` مرتبط بموظف
- طباعة PDF بخط Cairo

**الربط مع أقسام أخرى:**
- عند إنشاء فاتورة → يُخصَم `stock_qty` من المنتج تلقائياً
- عند تسجيل دفعة → يُحدَّث `paid_amount` ورصيد العميل
- يمكن إنشاء شحنة مباشرة من الفاتورة

#### ج) عروض الأسعار (Quotations)
**دورة حياة عرض الأسعار:**
```
مسودة → مرسلة → مقبولة (تحويل لفاتورة) / مرفوضة / منتهية الصلاحية
```

- بنود ديناميكية مع إمكانية إضافة/حذف صفوف
- تاريخ الصلاحية
- تحويل تلقائي لفاتورة مبيعات بزر واحد
- طباعة PDF

#### د) تقرير أعمار الذمم المدينة (Aging)
تحليل مديونية كل عميل حسب الفترات:

| الفترة | الوصف |
|--------|-------|
| 0 – 30 يوم | حالي |
| 31 – 60 يوم | تجاوز 30 يوماً |
| 61 – 90 يوم | تجاوز 60 يوماً |
| +90 يوم | متأخر جداً |

### 8 · 🚢 نظام التصدير (`export.js`)

**العملات المدعومة:**

**تقرير ربحية التصدير:**
- ربحية كل أمر تصدير
- مقارنة الإيرادات بالعملات الأجنبية مع التكاليف بالجنيه
- رسم بياني لتوزيع الأرباح حسب العملة أو الوجهة

### 9 · 🚛 اللوجستيك والشحن (`logistics.js`)

| نوع الشحن | بري / بحري / جوي |
| نقطة الانطلاق | المستودع المصدر |
| الوجهة | عنوان التسليم |
| السائق | اسم السائق |
| المركبة | رقم اللوحة ونوع المركبة |
| وزن الشحنة (طن) | |
| بوليصة الشحن | رقم Bill of Lading |

**التنبيهات التلقائية:**

### 10 · 💸 المدفوعات والمتحصلات (`payments.js`)

يُسجَّل كل تدفق نقدي في النظام هنا سواء كان:
- **متحصلة (Receipt):** من عميل
- **دفعة (Payment):** لمورد

**تفاصيل كل حركة:**

| الحقل | الوصف |
|-------|-------|
| النوع | متحصلة / دفعة |
| الطرف | اسم العميل أو المورد (`party_id` + `party_type`) |
| المبلغ | |
| التاريخ | |
| طريقة الدفع | نقدي / تحويل بنكي / شيك |
| رقم مرجعي | رقم التحويل أو الشيك |
| ملاحظات | رقم الفاتورة المسددة |

## 🔐 الأمان والصلاحيات — تفاصيل تقنية

### تشفير كلمات المرور
- كلمات المرور مُشفَّرة بـ **SHA-256** عبر `crypto.subtle.digest`
- تُخزَّن بالصيغة `sha256:hex_value`
- عند تسجيل الدخول تُقارن الهاشات وليس النصوص

### التوكن والجلسة
- التوكن يُشفَّر بـ **XOR** قبل حفظه في `sessionStorage`
- مفتاح التشفير مشتق من `userAgent` + `screen.width` (فريد لكل جهاز)
- مدة صلاحية التوكن: **8 ساعات**
- تسجيل خروج تلقائي بعد **8 ساعات** من عدم النشاط

### حماية تسجيل الدخول
- حد أقصى **5 محاولات** في 15 دقيقة
- عند تجاوز الحد: تجميد لمدة **15 دقيقة**
- رسالة خطأ تُحدَّد فيها الدقائق المتبقية

### تعقيم المدخلات (XSS)
- جميع الحقول النصية تمر عبر دالة `sanitize()` في `utils.js` قبل الحفظ
- حقول `password` و`email` مستثناة من التعقيم (تُعالَج بمنطق خاص)
- التحقق من الحدود الرقمية: المبالغ بين 0 و 10 مليار

### التحقق من سلامة البيانات
- عند بدء التطبيق، يفحص `DB.validateIntegrity()` جميع مفاتيح localStorage
- أي JSON تالف يُبلَّغ عنه في سجل الأخطاء `_errorLog`

### الصلاحيات (RBAC)

| الدور | الصفحات المتاحة |
|-------|----------------|
| **مدير عام / مدير** | جميع الصفحات بلا قيود |
| **محاسب** | لوحة التحكم، القيود، الحسابات، ميزان المراجعة، الشيكات، القيود المتكررة، إغلاق السنة، المدفوعات، المصروفات، مراكز التكلفة، تقارير P&L + الميزانية + الهالك + المخزون، الإعدادات |
| **موظف مبيعات** | لوحة التحكم، الفواتير، العملاء، الأعمار، عروض الأسعار، CRM، التصدير |
| **مدير مبيعات** | + المدفوعات، تقرير P&L، مديونية العملاء |
| **موظف مشتريات** | لوحة التحكم، فواتير الشراء، الموردون، الكتل، الألواح، المدفوعات |
| **موظف تصنيع** | لوحة التحكم، الكتل، مراحل التصنيع، الألواح، إدارة الجودة، تكاليف التصنيع |

## 🚀 تشغيل النظام محلياً

- لا حاجة لـ Node.js أو Python أو أي تبعيات

1. **استنسخ المستودع**
   ```bash
   git clone https://github.com/Yousefeid1/ERB-Marble.git
   ```

   ```bash
   # باستخدام Python
   python3 -m http.server 8080

   npx serve .

   # أو باستخدام PHP
   php -S localhost:8080
   ```
